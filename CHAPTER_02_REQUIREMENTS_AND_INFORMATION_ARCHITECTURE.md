# 第2章：CMS需求分析与信息架构设计

**预计学习时间：1.5 小时**

> 本章定位：
> - 从“会写页面/会写CRUD”过渡到“会设计系统”。
> - 先把需求和结构讲清楚，再进入后续技术实现。

---

## 本章目标

完成本章后，你应当能够：

1. 给出一个小型 CMS 的最小可用功能清单（MVP）。
2. 明确角色边界（管理员/编辑/访客）与权限差异。
3. 建立内容模型（文章、分类、标签、用户）及关系认知。
4. 产出前后台信息架构草图与核心业务流程说明。
5. 为后续数据库设计与 API 设计提供清晰输入。

---

## 2.1 为什么需求分析先于编码

很多初学者做项目时，容易陷入“先写再说”的路径：

- 页面先堆出来；
- 接口先写出来；
- 结果发现数据模型反复推翻。

在 CMS 里，需求和信息架构不清会直接导致：

- 数据表频繁重构；
- 前后台字段命名不一致；
- 权限逻辑后补，Bug 高发。

所以本章的核心不是写代码，而是**减少后续返工**。

---

## 2.2 定义最小可用版本（MVP）

我们先限定“第一个可交付版本”必须具备什么。

### 2.2.1 MVP 功能清单（建议）

建议把首版控制在 10 项左右：

1. 管理员登录/退出
2. 文章创建
3. 文章编辑
4. 文章删除
5. 文章列表（分页）
6. 草稿/发布状态切换
7. 分类管理
8. 标签管理
9. 前台文章列表展示
10. 前台文章详情展示

### 2.2.2 暂不纳入 MVP 的功能（建议）

- 评论系统
- 媒体库高级管理
- 站内搜索高级排序
- 多语言内容管理
- 审计日志与复杂审批流

这样做的目的：**把系统做完，比把系统做大更重要**。

---

## 2.3 角色与权限模型（最小版 RBAC）

本教程推荐先定义 3 类角色：

### 1) 访客（Visitor）
- 可访问前台公开内容；
- 不可访问后台。

### 2) 编辑（Editor）
- 可登录后台；
- 可新增/编辑文章；
- 通常不可管理系统级配置与用户。

### 3) 管理员（Admin）
- 拥有编辑权限；
- 可管理分类、标签、用户基础信息；
- 可进行发布控制与关键内容删除。

> 注意：即便是小型项目，也建议把“编辑”和“管理员”区分开，避免后期权限全耦合在一个角色里。

---

## 2.4 内容模型设计：核心实体与关系

小型 CMS 的核心通常围绕四类实体：

- `users`（用户）
- `posts`（文章）
- `categories`（分类）
- `tags`（标签）

以及一个关系表：

- `post_tags`（文章-标签，多对多）

### 2.4.1 建议字段（概念级）

#### users
- id
- username
- password_hash
- role
- created_at / updated_at

#### posts
- id
- title
- slug
- content
- status（draft/published）
- author_id
- category_id（可选）
- published_at（可选）
- created_at / updated_at

#### categories
- id
- name
- slug
- created_at / updated_at

#### tags
- id
- name
- slug
- created_at / updated_at

#### post_tags
- post_id
- tag_id

### 2.4.2 关键关系

- 一个用户可写多篇文章（users 1:n posts）
- 一个分类可归属多篇文章（categories 1:n posts）
- 一篇文章可有多个标签，一个标签可关联多篇文章（posts n:n tags）

---

## 2.5 前后台信息架构（IA）

IA（Information Architecture）关注“信息如何组织、页面如何被访问”。

### 2.5.1 后台 IA（建议）

- `/admin/login`：登录页
- `/admin/dashboard`：概览页（可选简化）
- `/admin/posts`：文章列表
- `/admin/posts/new`：新建文章
- `/admin/posts/:id/edit`：编辑文章
- `/admin/categories`：分类管理
- `/admin/tags`：标签管理

### 2.5.2 前台 IA（建议）

- `/`：首页（文章列表）
- `/posts/:slug`：文章详情
- `/categories/:slug`：分类页
- `/tags/:slug`：标签页（可选）

---

## 2.6 核心业务流程梳理（发布流程）

以“编辑发布文章”为例，流程可以定义为：

1. 编辑登录后台。
2. 新建文章并填写标题/内容/分类/标签。
3. 保存为草稿。
4. 二次编辑后切换为已发布。
5. 前台可见该文章。

这个流程会直接影响：

- `posts.status` 字段设计；
- 后台按钮与操作文案；
- 前台查询条件（只展示 published）。

---

## 2.7 非功能需求（小型项目版）

虽然我们不做企业级架构，但仍要有最基本的非功能要求：

1. **可维护性**：命名统一、目录清晰、文档可追溯。
2. **基础安全**：密码哈希、鉴权保护、输入校验。
3. **可部署性**：支持在单台 Linux 服务器稳定运行。
4. **可恢复性**：至少有基础数据备份策略。

## 2.8 本章产出清单（进入下一章前必须具备）

在进入第3章之前，你应当至少产出以下文档或草图：

- 一份 MVP 功能列表（10项左右）。
- 一份角色权限表（Visitor/Editor/Admin）。
- 一份实体关系草图（users/posts/categories/tags/post_tags）。
- 一份前后台路由草图。
- 一条“文章发布流程”的步骤说明。

如果这五项不完整，不建议进入数据库实现阶段。

## 2.9 本章小结

本章我们完成了从“写代码准备”到“系统设计准备”的关键过渡：

- 定义了 MVP，控制范围；
- 建立了角色权限边界；
- 设计了核心实体与关系；
- 规划了前后台信息架构与发布流程。

## 本章练习

### 练习1（MVP）
请写出你自己的 CMS 首版功能列表（8~12项），并说明每项是否必须在首版实现。

### 练习2（权限）
制作一张三角色权限矩阵表：

- 行：文章管理、分类管理、标签管理、用户管理、发布权限
- 列：Visitor / Editor / Admin

### 练习3（数据模型）
画出实体关系图（ER 简图），至少包含 `users`、`posts`、`categories`、`tags`、`post_tags`。

### 练习4（IA）
给出前后台路由结构，并标记每个路由对应的角色访问权限。

### 练习5（流程）
把“文章从创建到发布”的流程写成 6~8 个步骤，包含至少 1 个失败分支（例如校验失败）。

## 参考答案要点（自检）

### 练习1要点
- 功能范围可控；
- 首版聚焦核心闭环（创建-发布-展示）；
- 扩展功能明确延后。

### 练习2要点
- 访客只能读前台；
- 编辑可写内容但不碰系统级配置；
- 管理员拥有更高权限。

### 练习3要点
- `posts` 与 `users/categories` 是多对一；
- `posts` 与 `tags` 为多对多；
- 关系表字段命名清晰。

### 练习4要点
- 前后台路由前缀分离；
- 管理路由默认受鉴权保护；
- 页面与功能一一对应。

### 练习5要点
- 包含创建、草稿、发布、前台展示；
- 包含至少一个异常处理分支；
- 步骤顺序可执行、可验证。
